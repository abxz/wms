# 双组轮值制 v1.1

> **宪章**：交叉审不护短、互换改不藏私、上级审不失察、用户漏必追责。

## 〇、分组

| 组别 | 管辖 | 技术栈 | 边界 |
|:---:|:----|:------|:----|
| **A组（前端司）** | 前端 | React/TypeScript/Tailwind | `src/pages/*` `src/services/*` `src/components/*` |
| **B组（后端司）** | 后端 | FastAPI/Python/JSON存储 | `backend/modules/*` `backend/core/*` |

## 二、追责链（用户发现漏洞时）

```
🐛 用户发现漏洞
   │
   │  ┌─ 二审审核人 -8分  ← 最后一道关没守住
   │  ├─ 一审审核人 -5分  ← 交叉审没查出来
   │  └─ 开发组     -15分 ← 代码根因
   │
   ▼
🏛️ 皇城司记录归档
```

### 完整扣分表

| 场景 | 责任人 | 扣分 | 说明 |
|:----|:------|:---:|:----|
| 🐛 用户发现漏洞 | 开发组 | **-15** | 代码根因 |
| 🐛 用户发现漏洞 | 二审审核人 | **-8** | 审核失察 |
| 🐛 用户发现漏洞 | 一审审核人 | **-5** | 交叉没查出 |
| 👁️ 一审放过/二审查出 | 开发组 | **-5** | — |
| 👁️ 一审放过/二审查出 | 一审审核人 | **-3** | — |
| 🔄 互换修改仍不通过 | 互换双方 | **-12** | 倒逼认真改 |
| 🔥 熔断触发（第3次互换不过） | 互换双方 | **-15 + 冻结** | 军机处介入 |
| 💡 互换发现额外问题（确认后） | 发现者 | **+5** | 奖励发现新问题 |
| 💥 互换引入新问题（二审发现） | 修改者 | **-8** | 改坏了要负责 |
| 💥 互换引入新问题（用户发现） | 修改者 | **-15** | 承担开发者责任链 |

## 三、互换修改规则

### 范围限定

```
互换修改 = 只改二审明确指出有问题的代码行
  ✅ 修改指定文件、指定行号的问题
  ✅ 修改类型不匹配、字段名不一致等接口问题
  ✅ 标注 "extra: 发现额外问题"（经原组确认后 +5分）
  ❌ 不允许大规模重构/重写
  ❌ 不允许修改未指出的代码（除非标注 extra）
  ❌ 不允许设计变更/架构调整
```

### 多出的问题

```
互换修改时发现一审/二审没指出的额外问题：

发现者标注 "extra: 发现XXX问题"
  ├─ 经原组确认 ✅ → 发现者 +5分，原组不扣分
  └─ 错判（不是问题）→ 不计分

互换修改时改坏了代码（引入了新问题）：
  ├─ 二审发现 → 修改者 -8分
  └─ 部署后用户发现 → 修改者承担开发者责任链 -15分
```

### 熔断

```
第1次互换 → 二审不通过
第2次互换 → 二审不通过
第3次互换 → 🔥 熔断
  ├─ 军机处介入，裁决方案或裁定终止
  ├─ 互换双方各 -15分
  └─ 功能冻结，军机处终裁前不得继续
```

## 四、皇城司考核体系

### 双榜单（独立排名）

```
🏆【分组榜】               🏆【审核榜】
  名次  组别    积分           名次  角色     积分
  🥇   B组     92             🥇   二审甲    88
  🥈   A组     85             🥈   一审乙    76
  🥉   —       —              🥉   一审甲    72
```

考核人**独立排名**，与被审组彻底分开。审核人的分 = `发现问题数 × 3 - 放过问题数 × 10`，有动力严审不放水。

### 考核周期

| 周期 | 考核内容 | 输出 |
|:---:|:--------|:----|
| **日** | 扣分/加分流水 | 皇城司日志 |
| **周** | 各司积分变化 + 漏洞统计 | 周报 |
| **月** | 排名 + 升降级 + 趋势 | 月报 |

### 考核维度及权重

| 维度 | 权重 | 数据来源 |
|:----|:---:|:--------|
| 代码质量（用户发现漏洞数） | 40% | 皇城司追踪 |
| 审核质量（放过bug数） | 25% | 一审/二审记录 |
| 交付效率（按时完成率） | 20% | 尚书省记录 |
| 互换修改质量 | 15% | 互换后二审通过率 |

### 月报格式

```
🏛️ 皇城司·第X月考功奏章

┌──────────────────────────────────────┐
│  分组榜                               │
│  A组（前端司）     积分: 85  ━━ 排名🥇 │
│  B组（后端司）     积分: 80  ━━ 排名🥈 │
│                                      │
│  审核榜                               │
│  二审（都水司）     积分: 88  ━━ 排名🥇 │
│  一审组员甲         积分: 72  ━━ 排名🥈 │
│  一审组员乙         积分: 68  ━━ 排名🥉 │
│                                      │
│  本月统计                             │
│  代码提交: 28 次 / 一线审核: 22 次      │
│  发现漏洞: 3 个 / 二审拦截: 2 个       │
│  用户反馈: 1 个 | 熔断触发: 0 次       │
│                                      │
│  扣分TOP3                             │
│  ① A组·删除商品未确认       -5分       │
│  ② B组·API返回格式不符标     -3分       │
│  ③ 二审·漏审空指针路径       -8分       │
└──────────────────────────────────────┘
```

## 五、与其他朝廷技能的关系

| 朝廷技能 | 双组制中的映射 |
|:--------|:-------------|
| 三策制 | 决定项目分A/B组、任务拆分策略 |
| 超细Decree | 标注「前端」「后端」、文件路径、验证命令 |
| 派单制 | 按组派单、依赖编排 |
| 二驳制·一驳 | → 一审交叉审核（接口对齐审查） |
| 二驳制·二驳 | → 二审上级审核（整体审查） |
| 弹劾制 | 部署前最后一道防线（安全攻防） |
| 验讫令 | 部署验证（提供证据方可结案） |
| 皇城司 | 考核追责 + 双榜单排名 |
| 功过簿 | 积分记录 + 升降级 |
| 驳回熔断 | 互换修改第3次不过 → 熔断 |

## 六、铁律清单

| # | 铁律 | 违规后果 |
|---|------|---------|
| 1 | **一审必须交叉**，不允许自己审自己组 | 越权-5 |
| 2 | **一审范围限接口对齐**，不过问对方技术栈内部 | 二审驳回 |
| 3 | **二审不通过必须互换修改**，不可退回原组直接改 | 违规-5 |
| 4 | **互换修改限定点修改**，不允许大规模重构 | 二审驳回 |
| 5 | **用户漏洞先扣上级再扣开发**，顺序不可颠倒 | 皇城司追责 |
| 6 | **互换修改后必须重新二审**，不可直接部署 | 违规-10 |
| 7 | **考核双榜单独立排名**，审核人不与开发组混排 | 制度规定 |
| 8 | **同一模块第3次互换不通过触发熔断** | 军机处介入 |
| 9 | **扣分必须留痕**，无日志视为不扣 | 功过簿不计 |
| 10 | **互换发现额外问题可加分**，需原组确认 | 发现者+5 |
